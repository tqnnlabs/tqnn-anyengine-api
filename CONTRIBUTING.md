# Contributing

Thank you for your interest in contributing to TQNN.

The TQNN API is an actively developed project focused on providing a reliable, confidence-aware inference platform for structured data.

Whether you are reporting issues, improving documentation, or contributing code, your feedback is appreciated.

---

# Ways to Contribute

Contributions may include:

- Bug reports
- Documentation improvements
- Performance improvements
- Test coverage
- API usability feedback
- Example applications
- Feature suggestions

---

# Reporting Issues

When submitting an issue, please include:

- Operating system
- Python version (if applicable)
- API version
- SDK version (if applicable)
- Steps to reproduce
- Expected behavior
- Actual behavior
- Error messages or logs

Providing reproducible examples helps us investigate issues more efficiently.

---

# Pull Requests

Before opening a pull request:

- Verify that the code builds successfully.
- Follow the existing project style.
- Update documentation if behavior changes.
- Keep pull requests focused on a single improvement whenever practical.

---

# Documentation

Documentation improvements are always welcome.

Examples include:

- Clarifying API behavior
- Correcting examples
- Improving tutorials
- Expanding integration guides
- Fixing formatting or grammar

---

# Development Philosophy

TQNN is designed around several guiding principles:

- Clear and consistent APIs
- Confidence-aware inference
- Reliable developer experience
- Well-documented interfaces
- Stable public contracts

Contributions should support these goals whenever possible.

---

# Security

If you discover a security issue, please do not create a public issue.

Instead, contact TQNN Labs directly so the issue can be investigated and resolved responsibly.

Contact:

**tqnnlabs@gmail.com**

---

# Proprietary Components

This repository provides the public API for the TQNN platform.

The underlying TQNN runtime contains proprietary technology and is maintained separately.

Contributions should target the public API, documentation, testing, and developer experience unless otherwise discussed with TQNN Labs.

---

# Code Style

Please aim for:

- Readable code
- Meaningful variable names
- Consistent formatting
- Clear comments where appropriate
- Backward compatibility when practical

---

# Questions

If you have questions about integrating with the API or contributing to the project, please open a GitHub Discussion or contact TQNN Labs.

---

Thank you for helping improve TQNN.
